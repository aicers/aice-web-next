# Per-service draft TOML fixtures

This directory holds the **wire-format byte-compatibility fixtures**
that pin aice-web-next's per-service `toDraftToml` output against
aice-web's emitter at commit
`71c4623120dbb7ac35cb086c2f6c62c7f2df5372`. Each `*.toml` file is the
byte-for-byte output of Rust's `toml::to_string` over the matching
upstream config struct (`PigletConfig`, `GigantoConfig`, `HogConfig`,
`CrusherConfig`, `TivanConfig`) with the test inputs the
`src/__tests__/lib/node/services/*.test.ts` suite asserts against.

## Provenance

The fixtures are generated by `tools/draft-capture/` ‚Äî a small Rust
binary that depends on `toml = "0.8"` (the same crate version
aice-web pins) and replicates the upstream config struct definitions
from `aice-web/src/admin/node/settings/component.rs` verbatim,
including the full `ActiveModel` enum (every variant from base + the
non-gs additions). The current fixture inputs only construct a
two-variant subset of `ActiveModel` for the Hog "partial" case, but
the enum itself is reachable on demand ‚Äî extending a fixture to
cover any other model is a one-line edit to the corresponding
`hog_*()` builder. To regenerate after an aice-web struct change:

```sh
cargo run --manifest-path tools/draft-capture/Cargo.toml --release \
  -- src/__tests__/lib/node/fixtures
```

The binary writes one `.toml` file per case directly into this
directory, so the fixture diff that lands on the wire is exactly the
upstream serialiser's diff. With no path argument, it prints to
stdout instead ‚Äî useful for diffing without touching the tree.

This replaces the prior hand-authored fixture suite (which was
self-referential against the in-repo emitter and could not detect a
shared mistake between emitter and fixtures). The current suite is
generated by an emitter that does **not** share code with the TS
emitter, so the per-service tests now provide a real cross-emitter
regression guard.

### Why a Rust capture tool, not a live UI capture

The original sub-issue acceptance asked for fixtures captured by
driving aice-web's UI end-to-end and intercepting the GraphQL
`updateNodeDraft` payload. The Rust capture tool reaches the same
byte-level output (the GraphQL `draft` field is just
`toml::to_string(&config)?` ‚Äî see
`aice-web/src/admin/node/fetch.rs:181 / :733 / :806 / :843 / :865`)
without spinning up the Yew frontend, the GraphQL backend, or a
browser harness. Capturing through the UI exercises the same Rust
serializer through more layers; the bytes that land on the wire are
identical.

The end-to-end UI capture script remains a useful Phase Node-4/5
integration artefact (it would catch input-mapping bugs in the UI
itself), but it is not what guarantees byte-compatibility ‚Äî the
serializer is. The `tools/draft-capture/` binary is the canonical
reference for that contract.

## What the fixtures lock down

- **Field order** matches each upstream struct's declaration order
  (which is what `toml::to_string` emits, because serde's derive walks
  fields in source order). The TS emitters in
  `src/lib/node/services/*.ts` order their `TomlEntries` to match.
- **Quoting** uses TOML basic strings (`"‚Ä¶"`).
- **Empty-list cases** for fields subject to the asymmetric
  `all_checked_or_unchecked` rule
  (`aice-web/src/admin/node/fetch.rs:1665-1674`):
  - all checked ‚Üí field omitted (`None`)
  - partial ‚Üí strict subset (`Some([subset])`)
  - zero selected ‚Üí explicit empty array (`Some([])`)
- **Round-trip** ‚Äî the deserialiser hydrates each fixture back into a
  RHF state that re-serialises to the same string.

## Empty-list coverage

The asymmetric rule produces three distinct wire shapes; the suite
ships one fixture per shape for every list field that participates in
the rule:

- Piglet `protocols` / `dump_items` / `dump_http_content_types`
  (`sensor-{all-checked,zero-selected,partial}.toml`)
- Hog `active_protocols` / `active_models` / `active_sensors`
  (`hog-{all-checked,zero-selected,partial}.toml`)
- Hog `active_models` under the gs-cargo-feature rewrite
  (`hog-all-checked-gs.toml`) ‚Äî aice-web's `fetch.rs` rewrites a
  `None` (all checked) `active_models` back to `Some(Vec::new())` in
  gs builds, while `active_protocols` and `active_sensors` stay
  omitted. This is a fourth wire shape unique to gs mode; the
  capture tool emits it from a dedicated `hog_all_checked_gs()`
  builder so the byte-compat suite covers the rewrite end-to-end
  rather than only as a substring assertion in the TS emitter's own
  output.
